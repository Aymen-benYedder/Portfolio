What's the cheapest way to get started self-hosting LLMs in production?
Start with a single RTX 4090 (24 GB VRAM, ~$1,600) running quantized models via vLLM or Ollama. A 7B model at Q4 fits in ~5 GB and can serve internal tools at 40–80 TPS. For cloud, rent a single L4 or A100 on RunPod or Vast.ai ($0.50–$1.50/hr). Expect to handle 1–10 concurrent users on this setup before needing multi-GPU scaling.
When should I use vLLM vs Ollama vs SGLang for my project?
Use Ollama for local development, prototyping, and single-user tools (setup: 10 min). Use vLLM for production workloads serving 10+ concurrent users—it is the only engine with PagedAttention, continuous batching, and full speculative decoding support. Use SGLang as a vLLM alternative on GPU clusters where you need its native /generate path and YAML-based server configuration.
Can I self-host LLMs on CPU-only hardware?
Yes, but performance is 10–30× slower than GPU inference. A 7B model at Q4 on a modern 16-core CPU achieves roughly 2–5 TPS—usable for batch processing but not interactive applications. Tools like llama.cpp and Ollama support CPU inference with AVX2 and NEON optimizations. For interactive use, even a consumer GPU (RTX 3060 12GB) is strongly recommended.
How do I handle model updates without downtime in production?
Use Kubernetes with the vLLM Production Stack Helm chart. Deploy a new model version as a separate deployment, warm it up (15–60 seconds for model loading), then shift traffic via Envoy weighted routing. Implement canary releases with Flagger—route 10% traffic, monitor TTFT and error rate for 5 minutes, then gradual rollout. Always keep the previous model version as a rollback target.
What are the most important metrics to alert on for LLM inference?
Alert on: (1) TTFT p95 > 500ms—indicates prefill bottleneck or queue buildup; (2) KV cache usage > 85%—imminent OOM risk for new requests; (3) GPU memory bandwidth > 85%—true throughput bottleneck; (4) Error rate > 5%—model corruption or configuration issues; (5) Queue depth > 2× GPU count—need to scale horizontally. Start with these five; add GPU temperature and power draw for on-premise hardware.

By Aymen Ben Yedder — June 20, 2026

Self-hosting large language models in production has crossed the threshold from experimental curiosity to operational necessity. In 2026, open-weight models—Llama 4, DeepSeek V3.2 and V4, Qwen 3.5, Mistral Large—match or exceed GPT-4-class performance on most benchmarks, while inference engines like vLLM, SGLang, and Ollama have matured into production-grade serving stacks (Ollama hit 52M monthly downloads in Q1 2026). The decision to self-host versus consume APIs now hinges on three factors: token volume, data sovereignty requirements, and latency control. For DevOps engineers, the operational surface has expanded to include GPU cluster architecture on Kubernetes, real-time observability with Prometheus and Grafana, continuous batching via PagedAttention, and horizontal scaling through tensor and pipeline parallelism. This guide walks through every layer of the stack—hardware, engine selection, deployment patterns, monitoring, cost analysis, and security—drawing on real benchmarks and production-tested configurations.

Why Self-Hosted LLMs Now?

The market signals are unambiguous. Model API costs doubled to $8.4 billion in 2025 (PremAI, 2026), while 72% of companies plan to increase AI budgets this year and 44% cite data privacy as the top barrier to adoption (Kong 2025 Enterprise AI Report). Open-weight models have closed the quality gap: Llama 4 Scout via Groq API costs just $0.11 per million tokens, while DeepSeek V4 Flash runs at $0.30 per million input tokens with a 1-million-token context window (AgentDeals, June 2026). Meanwhile, enterprise AI workloads are increasingly governed by regulations ranging from GDPR and HIPAA to Tunisia's LOPD 2004-63, which mandates data localization and explicit consent for processing. Self-hosting is no longer just a cost play—it is a compliance and sovereignty play.

The Inference Engine Decision

The single most consequential technical decision you will make is which inference engine to run. The gap between engines is not marginal—it is an order of magnitude.

vLLM: The Production Standard

vLLM delivers 793 tokens per second on identical hardware where Ollama manages 41 TPS—a 19× throughput gap at scale (Jaipal Singh, 2026 benchmarks). At 128 concurrent users, vLLM maintains sub-100ms P99 latency; Ollama spikes to 673ms under the same load. This performance comes from three architectural innovations: PagedAttention, which reduces memory fragmentation by over 40%; continuous batching, which dynamically groups incoming requests into optimal batches; and native speculative decoding with EAGLE, MTP, and PARD methods delivering 2–4× speedups (PARD achieves up to 4.87× acceleration, reaching 381.7 tokens/sec on Qwen2.5 7B).

vLLM also exposes 20+ Prometheus metrics via its /metrics endpoint with a vllm: prefix, covering request counts, KV cache usage, TTFT histograms, TPOT histograms, and end-to-end latency distributions (vLLM Metrics Design). For any deployment serving 10 or more concurrent users, vLLM is the default choice.

When Ollama Makes Sense

Ollama dominates developer mindshare—95K+ GitHub stars as of June 2026 and 52M monthly downloads—and its 10-minute setup is unmatched for prototyping (GitHub). But it lacks PagedAttention, tensor parallelism, multi-node support, and performs poorly under concurrent load. Use Ollama for local development, single-user tools, and quick experimentation. Do not use it for production workloads.

SGLang and TGI

SGLang (8K+ stars) provides a native /generate path and offline Engine for in-process batch work, making it a strong vLLM alternative on GPU clusters. Hugging Face's TGI, meanwhile, entered maintenance mode in December 2025 (Hugging Face). It is not recommended for new deployments.

Hardware: Sizing GPUs for Your Model

The VRAM equation is straightforward: roughly 2 GB per 1 billion parameters at FP16, and ~0.5 GB per 1B at INT4 quantization (Spheron Network). A 7B model requires 14 GB at FP16 or roughly 3.5 GB at INT4. A 70B model demands 140 GB at FP16 or about 35 GB quantized—requiring two A100s or H100s for production serving. The KV cache adds 20–40% overhead: at 32K context with a batch of 8, the cache alone consumes 40+ GB on a 70B model (Inferbase GPU Sizing Guide).

For production, the GPU tier ladder in 2026 looks like this: enterprise deployments target the B200 (FP4 Tensor Cores, ~15× faster inference than H200), H200 (141 GB HBM3e at 4.8 TB/s), and H100 (3,958 TFLOPS at FP8). Professional tiers use the L40S or RTX 6000 Ada (48 GB each). The consumer standout is the RTX 5090 with 1.8 TB/s bandwidth and native NVFP4 support. On the AMD side, the MI300X delivers 192 GB HBM3 at 5.3 TB/s and 1,307 TFLOPS FP16—a legitimate alternative if you want to avoid NVIDIA lock-in.

Beyond GPUs, budget for 32–64 GB system RAM for KV cache offloading, an NVMe SSD with 100–500 GB free for model files, and 40 Gbps minimum networking between GPU nodes if you plan to use tensor parallelism.

Architectural Patterns That Scale

Disaggregated Prefill and Decode

This is the single most impactful architectural decision you can make. Colocating prefill (compute-bound, processes the prompt) and decode (memory-bandwidth-bound, generates tokens one by one) on the same GPU causes throughput collapse under high concurrency. Separating them into dedicated GPU pools—prefill GPUs sized for compute throughput, decode GPUs sized for memory bandwidth—with a 40 Gbps+ interconnect prevents p50 latency from exceeding 200ms (Markaicode GPU Cluster Architecture).

Parallelism Strategies

You have five levers for scaling across GPUs. Tensor parallelism splits individual layer matrices across GPUs and delivers the lowest latency—roughly 3× TTFT improvement—but requires NVLink or 40 Gbps+ interconnect. Pipeline parallelism splits layers across GPUs and works well for models that do not fit on a single GPU, delivering 2.5–3× TTFT reduction at high concurrency with moderate interconnect requirements. Data parallelism replicates the full model across GPUs and splits incoming requests, yielding roughly 50% throughput gain at moderate concurrency. Expert parallelism targets mixture-of-experts models like Mixtral and DeepSeek by distributing experts across GPUs. Context parallelism splits long sequences across GPUs for 128K+ token context windows.

Speculative Decoding

Speculative decoding methods in vLLM reduce inter-token latency by having a lightweight draft model propose candidate tokens that the target model then verifies in parallel. EAGLE delivers 2–3× speedup with a draft-model approach. MTP (Multi-Token Prediction) reaches 2–4× using an internal drafter head. PARD achieves 1.78–4.87× acceleration at low cost via parallel draft model adaptation (arXiv 2504.18583v3). Even simple n-gram speculative decoding yields 1.2–1.5× with zero additional model overhead.

Kubernetes Deployment: The Production Playbook

Deploying vLLM on Kubernetes is the reference architecture for production self-hosted inference. The vLLM Production Stack provides a complete Helm chart with autoscaling (HPA based on vllm:num_requests_waiting and vllm:kv_cache_usage_perc), GPU scheduling with node selectors, Pod Disruption Budgets, Persistent Volume Claims for model weights, and ConfigMap-driven configuration.

A production values file configures the model, replica count, GPU requirements, tensor parallel size, and autoscaling thresholds. The HPA should scale on inference-specific metrics, not CPU or memory—traditional scaling rules fail with LLM cold starts lasting 15–60 seconds for model loading. Set queue depth triggers: scale when waiting requests exceed 2× the GPU count.

For the broader Kubernetes ecosystem, integration options include KServe for serverless inference, Kaito for automated GPU node provisioning, NVIDIA Dynamo for multi-node inferencing, and Flagger or Argo Rollouts for canary deployments during model updates.

Your CI/CD pipeline for model updates should follow a GitOps workflow: Git push triggers GitHub Actions or GitLab CI, which runs benchmark evaluations, containerizes the model, pushes to a registry, then syncs via ArgoCD or Flux. The rollout uses canary traffic shifting via Envoy—route 10% of requests to the new model version, monitor TTFT and error rate for 5 minutes, then proceed with the full rollout. Always maintain the previous version as a rollback target.

Observability: What to Measure and Alert On

LLM inference observability requires metrics beyond standard application monitoring. The five critical metrics are:

  • TTFT (Time to First Token) — latency from request receipt to first token generation. Alert at p95 > 500ms. This indicates a prefill bottleneck or queue buildup.
  • TPOT (Time Per Output Token) — inter-token latency during the decode phase. Alert at p95 > 100ms.
  • KV Cache Usage — the percentage of KV cache blocks consumed. Alert at > 85%. This is the hidden memory consumer; at 32K context with batch 8 on a 70B model, the cache alone adds 40+ GB.
  • GPU Memory Bandwidth Utilization — the true throughput bottleneck. When utilization exceeds 85%, latency spikes follow (Markaicode).
  • Queue Depth — the number of waiting requests. When it exceeds 2× the GPU count, it is time to scale horizontally.

The vLLM v1 engine exposes all of these as Prometheus metrics. Set up a Prometheus scrape job targeting the vLLM /metrics endpoint, then use Grafana dashboard ID 25263 (vLLM Metrics, updated May 2026) for visualization. Add DCGM Exporter for NVIDIA GPU-level metrics, OpenTelemetry for distributed tracing across the inference chain (--otlp-traces-endpoint and --collect-detailed-traces), and Alertmanager rules for TTFT, queue depth, and out-of-memory conditions.

Cost Analysis: When Self-Hosting Makes Financial Sense

Self-hosting breaks even at roughly $20K per month in API spend or ~100M tokens per month (TokenMix). Below 10M tokens per month, APIs are cheaper by a wide margin. Between 10M and 100M tokens, a hybrid approach works best. Above 100M tokens per month, self-hosting saves 40–60% compared to equivalent API consumption.

Running a 70B model on 2× A100 80GB GPUs in the cloud costs $3,000–$5,000 per month for GPU rental alone. Adding storage, networking, and ongoing engineering maintenance (5–10 hours per month), total operational costs land at $3,725–$7,600 per month. The equivalent API cost at 100M+ tokens per month runs $8,000–$15,000 (TokenMix). The first month includes 20–40 hours of engineering setup, adding $1,500–$6,000 in one-time costs.

But teams consistently underestimate three hidden costs. First, actual GPU utilization averages 20–40% for most inference workloads—you are paying for compute you do not use. Second, initial deployment timelines run 12–16 weeks, not the 8 weeks most teams plan for—a 200% overrun. Third, GPU and inference specialists command a 20–30% salary premium over general DevOps engineers. Budget for these before committing to self-hosting.

Security: The Non-Negotiable Baseline

Self-hosted inference introduces a threat surface that includes prompt injection, model weight exfiltration, unauthorized inference access, GPU resource exhaustion, and model jailbreaking. The baseline mitigations are well-established:

  • Never expose inference ports directly. Always terminate TLS at a reverse proxy (NGINX or Envoy) with rate limiting. Configure token-aware rate limiting: 10 requests per second per client with a burst of 5, and 5 concurrent connections per client.
  • Authenticate every request. JWT authentication with scoped claims, short expiration, and separate roles for inference consumers, prompt engineers, model administrators, and auditors.
  • Segment your network with mTLS. Isolate inference, vector database, and orchestrator networks. This prevents a compromised service from accessing model weights or RAG data.
  • Encrypt model weights at rest on read-only mounted volumes with restricted file system permissions. Verify checksums and signatures before loading.
  • Harden containers. Drop all capabilities, run as non-root, apply seccomp profiles, and never use privileged mode.
  • Audit request metadata (not prompt content) for compliance with GDPR, HIPAA, SOC 2, and EU AI Act requirements.

For Tunisian readers specifically: LOPD 2004-63 requires data localization, explicit consent for processing, and a maintained processing registry. Self-hosting on local infrastructure is often the only way to comply with these requirements when processing sensitive data through LLMs.

The DevOps Toolchain

A production self-hosted LLM stack integrates into the existing DevOps toolchain at every layer. ArgoCD and FluxCD handle GitOps sync for model deployment manifests. Terraform or OpenTofu provisions GPU nodes, VPCs, and firewall rules. HashiCorp Vault or External Secrets manages Hugging Face tokens, API keys, and model registry credentials. Flagger or Argo Rollouts orchestrates canary deployments for model updates. ELK or Loki handles inference request logging and audit trails. PagerDuty or OpsGenie routes alerts for SLO breaches. And a model registry on S3-compatible storage (MinIO, Cloudflare R2) with versioned paths, checksum-verified references, and automated benchmark gates ensures that no model version ships without passing evaluation thresholds.

Conclusion

Self-hosting LLMs in production is operationally viable in 2026, but it demands a new layer of DevOps capability. The stack spans hardware provisioning, inference engine configuration, Kubernetes orchestration with GPU-aware scheduling, real-time observability of KV cache and TTFT metrics, and security controls that address a threat model specific to AI workloads. The financial case is clear above 100M tokens per month, and the compliance case is clear wherever data sovereignty regulations apply.

Start small. A single RTX 4090 running a quantized 7B model via vLLM will serve internal tools at 40–80 TPS and teach you the operational patterns. From there, the path to multi-GPU, multi-node production inference is a matter of scaling what you have already learned—adding tensor parallelism, disaggregating prefill and decode, instrumenting with Prometheus and Grafana, and tightening the security baseline. The tools are production-ready. The models are competitive. The rest is DevOps.
